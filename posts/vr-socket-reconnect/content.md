# VR看房：Socket断线重连下的状态一致性实战

## **背景**

去年我负责一个**线上VR带看房**项目。核心场景是：房产顾问与用户通过Web端进行实时语音 + 空间同步，顾问可以在3D户型图中移动、用激光笔圈点重点区域，用户的屏幕实时跟随顾问的视角，同时双方可以进行文字/语音对话。

这个项目对实时性要求很高，所有非音视频的信令（视角同步、画笔轨迹、房间切换等）都通过 **WebSocket** 承载。音视频则走独立的WebRTC通道。

项目上线后，后台收到不少用户反馈："顾问刚指的那个位置我没看到"、"我切换了一下网络就看不到顾问的操作了"、"画面卡住但是声音还在"。深入排查后，我们发现问题的根源直指**Socket断线重连时的状态丢失**。

本文分享我们如何一步步分析并解决这个问题，最终将弱网环境下的消息丢失率从 8% 降到 0.6% 以下。

## **技术栈概览**

- **实时信令通道**：原生 WebSocket（生产环境使用 `socket.io-client` v4 做封装，但核心机制自行实现）
- **消息协议**：JSON over text，每个消息体包含 `type`, `payload`, `timestamp`, `seqId`
- **心跳机制**：业务层心跳（`ping/pong` 带时间戳），底层 TCP keep-alive 辅助
- **状态管理**：Vuex（主线程）+ Web Worker（负责 Socket 收发与队列管理）
- **重连策略**：指数退避 + 最大重试次数限制

## **场景还原：VR带看房中的实时协作**

一次典型的带看流程如下：

1. 顾问创建带看房间，生成分享链接。
2. 用户点击链接进入，WebSocket 建立连接，双方加入同一个 Room。
3. 顾问在 3D 场景中移动（位置变化、朝向变化、缩放变化），客户端每隔 100ms 采样一次，发生变化后发送 `view_update` 消息。
4. 顾问使用**激光笔**在户型图上圈画重点（如"这里是承重墙"、"橱柜是赠送的"），每次笔触产生多个 `stroke_point` 消息。
5. 用户端实时渲染出顾问的虚拟形象位置和笔迹。
6. 双方可以打字提问、发送语音片段（走另一通道）。

在整个过程中，Socket 连接维持着所有"同步状态"。一旦断线，**用户端就像凭空消失了一段时间**，顾问发出的所有指令都无法送达，用户重连后看到的是一个完全没有过渡的"跳跃"画面，完全不知道刚才发生了什么。

## **核心痛点：断线重连带来的状态黑洞**

### **典型场景**

假设顾问正在给用户讲解厨房的布局：

- `t=0s`：顾问发送第一条笔画 `{type:"stroke", points:[...], color:"red"}`
- `t=1s`：用户走出电梯，网络从 Wi-Fi 切换到 4G，Socket 连接中断（TCP 层面不会立刻感知）
- `t=1.5s`：顾问继续画了第二条笔画
- `t=2s`：顾问切换视角到卧室，发送 `view_jump` 消息
- `t=2.5s`：用户的网络恢复，Socket 自动重连成功
- `t=3s`：用户端只显示了顾问此刻的位置（卧室），**厨房的两条红色笔画和视角切换过程全部丢失**

用户可能会说："您刚才说的厨房改动在哪里？我没看到啊。" 顾问不得不重新操作一遍，体验极差。

### **为什么普通重连不够用？**

- **TCP 断线检测延迟大**：默认需要几十秒甚至分钟级才能发现连接已死，用户不可能等那么久。
- **WebSocket 自带的重连（如 Socket.IO）只恢复连接，不恢复业务状态**：它不知道用户之前收到了哪些消息，也不知道哪些消息在断线期间丢失了。
- **消息发送是"发后即忘"**：没有确认机制，服务端也不会为离线客户端缓存消息（为了性能，默认不缓存）。

这就导致了：**连接恢复 ≠ 状态恢复**。

## **解决方案：一套面向业务状态的重连同步协议**

我们的方案可以概括为：**主动心跳 + 消息序列号 + 断线缓存 + 有序重放**。

### **1. 业务层主动心跳，快速感知断线**

底层 WebSocket 的 `onclose` 事件触发太慢（尤其在弱网下）。我们实现了一个业务心跳：

- 客户端每隔 `5s` 发送一条 `{type:"heartbeat", clientTime: Date.now()}`。
- 服务端收到后立即原样返回 `{type:"heartbeat", serverTime: Date.now()}`。
- 客户端维护一个 `heartbeatMissCount`：如果连续 **3 次** 发送后没有收到对应回包（超时 3s），主动判定连接失效，立即触发重连。

这样断线感知时间从可能的 30s 压缩到了 `5s * 3 = 15s` 以内。大部分场景下，用户甚至感觉不到已经重连过一次。

### **2. 消息序列号（seqId）—— 状态同步的基石**

我们为**每一条**从客户端发出的消息（以及服务端广播的消息）都增加两个字段：

```json
{
  "type": "stroke",
  "seqId": 10086,
  "timestamp": 1620000000000,
  "payload": { ... }
}
```

- `seqId` 是一个全局递增的 64 位整数，由**服务端生成并下发**。客户端发送消息时先请求服务端分配 seqId（也可以客户端生成，但要求所有客户端严格递增，通常用服务端颁发）。
- **为什么不用客户端生成？** 多客户端之间无法保证 seqId 唯一且单调递增，服务端颁发可以统一全局顺序。

实际我们做了优化：客户端本地预先生成一个临时 id，服务端回包时映射到正式 seqId。但为了简化，这里描述最终方案：**所有消息的 seqId 由服务端单调递增并存储到 Redis 中，每个 Room 独立维护**。

### **3. 客户端消息队列与 ACK 机制**

- 客户端发出的每条消息，都会暂存在本地 `pendingQueue` 中，并启动一个定时器等待服务端的 `ack`（确认消息）。
- 服务端收到消息后，除了广播给 Room 内其他人，还会单独给发送方回复一条 `ack:{seqId}`。
- 客户端收到 `ack` 后，从 `pendingQueue` 中移除该消息。若 3s 内未收到 ack，则触发重传（最多 3 次）。

这保证了**关键信令**不会因为网络抖动而永久丢失。

### **4. 断线重连后的状态同步流程**

这是整个方案中最关键的一环。

当客户端重连成功后（新的 WebSocket 连接建立），它立即发送一个 `sync` 消息：

```json
{
  "type": "sync",
  "lastSeenSeqId": 12345   // 客户端本地最后一条成功处理的服务端消息的 seqId
}
```

服务端收到后，从消息历史记录中（我们使用 Redis Stream 或内存环形缓冲区，保留最近 5 分钟的所有消息）查询所有 `seqId > lastSeenSeqId` 且属于该 Room 的消息，打包批量下发给客户端。

客户端收到这批历史消息后，**不直接渲染**，而是按照 seqId 顺序重新"执行"一遍：

- 对于 `stroke` 消息：清空当前笔画缓存，重新绘制笔画轨迹（快速回放，用时 0.5 秒内完成）。
- 对于 `view_update` 消息：不瞬移，而是用动画平滑过渡到最终位置。
- 对于 `room_change` 消息：直接跳转。

为了避免回放期间用户手动操作造成冲突，我们在"同步中"状态禁用用户的视角控制输入，并显示提示："网络恢复，正在同步顾问的讲解进度..."。回放结束后，状态自动恢复可交互。

### **5. 状态机驱动的连接管理**

我们设计了一个明确的状态机，让 UI 和业务逻辑能够清晰应对各种情况：

```text
[INIT] -> [CONNECTING] -> [CONNECTED] <-> [RECONNECTING] -> [CLOSED]
```

- `RECONNECTING` 状态时：
    - 界面顶部出现"网络波动，努力重新同步..."的横幅（非模态，不打断俯瞰浏览）。
    - 停止发送非关键的信令（例如定期上报的"用户是否在注视某点"这类非必需消息）。
    - 维护一个 `offlineQueue`，新产生的消息（如用户打字提问）暂存，等到 `CONNECTED` 时再发送。
    - 如果重连成功且历史消息同步完成，清空 `offlineQueue` 按序发送。
- 若重连失败（超过最大重试次数），则提示用户"连接已断开，点击重新加入"，并提供一键重入房间的功能。

### **6. Web Worker 保活（高级优化）**

浏览器限制后台标签页的定时器和网络请求，但 Web Worker 相对独立。我们将 Socket 的核心接收/发送逻辑放在一个 Worker 中，主线程只负责 UI 渲染和用户输入。

- Worker 持续维持心跳和接收历史消息，主线程进入后台时可以通过 `postMessage` 暂停非关键的同步。
- 当用户切回前台时，Worker 里已经缓存了断线期间的消息（如果有），主线程唤醒后一次性获取并回放。

这个优化在移动端（尤其是 iOS Safari）效果明显，大大减少了因切后台导致的"假掉线"问题。

## **最终效果**

方案上线后，我们采集了两周的数据对比：

| **指标** | **优化前** | **优化后** |
| --- | --- | --- |
| 弱网场景消息丢失率 | ~8% | <0.6% |
| 断线重连平均耗时 | 12s | 2.7s |
| 用户重连后状态不一致投诉 | 17次/周 | 1次/周 |
| 心跳探测额外流量开销 | 0 | <1KB/分钟 |

更重要的是，用户反馈中"看不懂顾问动作"的负面评价显著减少，带看完成率提升了约 11%（业务侧数据）。

## **总结与思考**

通过这个项目，我深刻体会到：**实时应用不能只保证"连接"，更要保证"状态"**。断线重连不仅仅是重新拉起一个 TCP 连接，而是要设计一套业务层的同步协议，让客户端和服务端能够"对账"。

一些可以复用的设计原则：

- **全局序列号（seqId）** 是分布式同步的基础。
- **业务心跳** 优于底层心跳，能更快感知真实可用性。
- **历史消息缓存** 不需要存太久（5 分钟足够），但要保证可靠性和顺序。
- **UI 需要明确的状态反馈**：让用户知道"正在恢复"而不是"卡死了"。
- **区分消息优先级**：关键信令（圈画、房间切换）需要 ACK 与重传，非关键信令（实时位置微调）可以直接丢弃。

未来我们计划引入 **CRDT** 思想来处理更复杂的冲突（比如双方同时操作户型），但目前这套方案已经稳定支撑了日均千次带看通话。
